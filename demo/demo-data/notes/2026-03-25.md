## Caching Research

Spent the morning researching caching strategies for our API layer. We're seeing p99 latencies of ~800ms on the user listing endpoints and it's becoming a bottleneck.

### Options Evaluated

**Redis**
- Pros: mature, great ecosystem, pub/sub for invalidation, built-in TTL
- Cons: another service to operate, need to handle connection failures
- Fits our use case well for session + query caching

**Memcached**
- Pros: simpler, slightly faster for pure key-value
- Cons: no data structures, no persistence, harder to do cache invalidation
- Better for simple caching but we want more flexibility

**Application-level (in-memory)**
- Pros: zero latency, no network hop, simple
- Cons: doesn't scale across instances, memory pressure, cold start after deploy

### Decision

Going with **Redis**. Reasons:

1. We already use it for job queues (Sidekiq)
2. Need pub/sub for distributed cache invalidation
3. The Lua scripting support is useful for atomic cache-aside patterns

Wrote up the full analysis in [[doc:caching-strategy]].

### Prototype

Quick prototype using `ioredis` with our existing Express setup:

```typescript
import Redis from "ioredis";

const cache = new Redis(process.env.REDIS_URL);

async function cachedQuery<T>(
  key: string,
  ttlSeconds: number,
  fetcher: () => Promise<T>
): Promise<T> {
  const cached = await cache.get(key);
  if (cached) return JSON.parse(cached);

  const result = await fetcher();
  await cache.setex(key, ttlSeconds, JSON.stringify(result));
  return result;
}
```
